# IT Solution Lab 面试讲解

这份材料用于在面试中说明项目的目标、取舍和当前边界。项目内容以现有代码和学习记录为准，不把模板方案、个人实验或待验证项描述成真实客户交付结果。

## 30 秒项目介绍

IT Solution Lab 是一个面向企业 IT 解决方案学习的个人技术实验室。我从售后工程师向安全售前方向发展，希望把网络、安全、云计算和方案设计的学习过程整理成可以持续复盘的作品集。

项目把技术地图、产品学习模板、个人实验记录、方案模板、厂商场景分析和成长看板连接起来，表达一条从技术学习、产品理解、实验验证到方案设计和选型思考的能力形成链路。当前内容以学习材料和个人验证记录为主，不代表商业产品或真实客户项目。

## 3 分钟项目介绍

### 为什么创建

售后工作通常更关注现网问题定位、配置实施和故障处理。向安全售前发展后，还需要能够从客户问题出发，解释需求约束、技术能力、产品组合、架构关系和实施边界。IT Solution Lab 用一个可持续扩展的静态站点记录这段能力迁移过程，而不是只做产品名称的展示页。

### 系统如何组织

- **Technology Map**：按基础能力、网络、安全、云计算和解决方案建立学习顺序与技术关系。
- **Product Lab**：以统一模板整理产品定位、客户问题、核心能力、配置实践、故障排查和售前关注点。当前包含 6 个产品学习模板。
- **Labs**：记录“企业出口安全模拟”这一条个人实验，围绕目标、环境、拓扑、配置方向、验证项和复盘方法组织内容。
- **Solutions**：维护“成长型企业安全与云基础设施升级”这一模板方案，连接客户问题、架构、产品能力、验证准备和实施规划。它目前正在完善，不是已交付案例。
- **Vendor Comparison**：对深信服、华为和 H3C 做场景化选型分析，比较产品定位、技术路线、适用场景和限制因素，不做厂商排名。
- **Dashboard**：聚合技术、产品、实验、方案和选型内容，展示能力证据如何逐步形成，而不是输出未经依据的能力评分。
- **Resources / Business Tools**：提供设备知识库、手册、交付与排障资料，以及工单、报价等个人工作记录入口。

### 能力形成链路

```text
技术学习 → 产品理解 → 实验验证 → 厂商分析 → 方案设计 → 售前流程 → 能力证据
```

这条链路是页面之间的引用关系：方案可以引用实验的验证范围，Dashboard 聚合各模块已有记录，但不会复制或虚构一套新的业务数据。

## 为什么选择这个技术方案

### 原生 HTML、CSS 和 JavaScript

项目以原生 HTML、CSS 和 JavaScript 为主，没有引入前端框架或运行时依赖。对于一个个人学习实验室，页面数量有限、数据规模可控，原生方案便于直接理解 DOM、事件、状态和构建过程，也减少框架升级和依赖维护成本。

### 静态部署

站点通过 Node.js 构建脚本生成静态资源，并部署到 Vercel。静态部署适合当前以文档、筛选和详情展示为主的工作负载，不需要为尚未存在的账号体系、服务端数据库或多人协作提前设计复杂后端。

### Hash 路由

页面使用现有的 Hash 路由在模块之间切换，例如 `#technology`、`#products`、`#labs` 和 `#dashboard`。这样在静态托管环境中可以直接访问带 hash 的页面，不依赖服务端为每个路径配置回退规则，同时保留浏览器前进、后退能力。

### 数据驱动渲染

产品、实验、方案和厂商分析以项目内受控数据为基础，由页面逻辑动态生成统计、筛选结果和详情内容。模块之间通过稳定的 ID 和引用关系复用数据，避免首页或 Dashboard 维护重复列表，也让状态和数量可以从真实数组计算。

## 最大挑战

### 数据一致性

同一个实验、方案或产品会出现在首页、详情页和 Dashboard。如果每个页面单独写一份名称和状态，很快会出现“企业出口安全模拟”或方案状态不一致的问题。因此项目优先使用已有数据源和稳定 ID，展示层再统一中文状态表达。

### 状态管理

产品学习状态、实验状态和方案生命周期并不完全相同。例如实验可以处于“进行中”并且模板已建立，方案可以处于“正在完善”但架构已经建立。项目保留这些业务语义，避免用一个百分比或评分字段强行表示所有模块的完成度。

### 内容真实性

这是作品集项目最需要控制的边界。实验验证项只描述验证目标、检查方式和预期观察结果；方案是模板方案；厂商对比是场景化学习材料。没有真实客户名称、预算、上线结果、验收结果、性能数字或市场排名时，就明确标记为待补充或待验证，而不是为了完整感编造事实。

### 模块关联

售前能力不是产品、实验和方案的简单相加。项目需要表达客户需求如何进入技术选型，产品能力如何进入架构，架构如何关联验证准备，最后在 Dashboard 中形成可检查的证据链。为此，Solutions 使用实验引用关系，Dashboard 聚合既有模块，而不是复制 Labs 或 Product Lab 的数据。

## 关键技术决策

| 决策 | 选择 | 原因 |
| --- | --- | --- |
| 页面实现 | 原生 JS + HTML/CSS | 依赖少，便于理解交互与内容结构，适合个人静态项目 |
| 部署方式 | 静态构建 + Vercel | 当前以知识内容和交互展示为主，部署链路简单且可公开预览 |
| 页面导航 | Hash 路由 | 兼容静态托管，支持直接访问模块 hash 和浏览器历史 |
| 内容组织 | 数据驱动渲染 | 统一产品、实验、方案和厂商数据来源，减少重复维护 |
| 本地记录 | Browser Local Storage | 保留个人工单、报价和手册等浏览器内记录，不假设已有账号或云端同步 |
| 可访问性 | 原生语义、标签和键盘焦点 | 在不引入框架的情况下改善导航、筛选和弹窗的可用性 |

## 当前不足

- 当前只有一条“企业出口安全模拟”个人实验记录，真实实验积累和可复查证据仍需增加。
- 实验中的验证项主要是验证准备和记录框架，尚未形成完整的配置、日志截图和实际测试结论。
- “成长型企业安全与云基础设施升级”是正在完善的模板方案，不代表真实客户交付，需求约束、容量、授权、兼容性和实施边界仍需结合具体环境验证。
- 产品学习内容需要持续补充官方资料来源、版本边界和更多个人实践记录。
- 厂商对比用于练习客观选型，不能替代正式的厂商资料核对、PoC 或客户环境测试。
- 当前没有账号体系、云端协作、服务端数据持久化和完整自动化端到端测试。

## 面试时需要主动说明的边界

这个项目展示的是工程师如何组织学习、实践和方案思考，而不是客户项目交付证明。数量、状态和页面统计只代表当前记录，不等于能力评分；模板方案、个人实验和厂商分析也不应被解读为真实项目结果。后续工作的重点是增加可复查的个人实验记录，补充官方资料依据，并在明确环境约束后逐步完善方案验证。
