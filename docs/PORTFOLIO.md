# IT Solution Lab 项目案例

## 项目背景

IT Solution Lab 起源于一名深信服售后工程师的职业转型学习需求。售后工作更关注产品部署、故障定位和交付过程，而安全售前需要进一步理解客户问题、技术路线、产品组合、方案边界与验证方法。

这个项目用于把分散的学习笔记整理成一条可持续积累的路径：从网络、安全和云计算基础出发，通过产品理解、实验验证、厂商分析和方案设计，逐步建立面向安全售前工作的分析与表达能力。

## 项目定位

IT Solution Lab 是：

- 个人技术学习实验室，用于结构化整理企业 IT 知识。
- 售前能力积累平台，用于练习从客户问题到技术方案的思考过程。
- 个人成长档案，用于记录当前内容、状态和后续完善方向。
- 非商业产品，不提供生产环境服务或正式技术承诺。
- 非真实客户项目，不代表已经完成客户交付、上线或验收。

项目重点不是展示产品参数数量，而是说明一个工程师如何把学习内容逐步转化为可检查、可关联、可继续完善的能力证据。

## 当前系统结构

| 模块 | 当前作用 | 形成的证据 |
| --- | --- | --- |
| Technology Map（技术地图） | 整理网络、安全、云计算和解决方案四个学习方向及推荐顺序 | 技术领域、学习状态与后续入口 |
| Product Lab（产品实验室） | 以统一模板整理产品定位、客户问题、核心能力、实践方向、排障思路和售前关注点 | 6 个深信服产品学习模板 |
| Labs（实验中心） | 按目标、环境、配置方向、验证项和复盘方法记录个人实践 | “企业出口安全模拟”个人验证记录 |
| Solutions（解决方案中心） | 练习从客户问题、约束、架构和产品组合到实施规划的方案表达 | “成长型企业安全与云基础设施升级”模板方案 |
| Vendor Comparison（厂商对比） | 从场景、技术路线、优势方向、限制因素和选型条件分析厂商 | 深信服、华为、H3C 的场景化学习材料 |
| Dashboard（成长能力看板） | 聚合已有模块的数量、状态和引用关系 | 从技术学习到选型分析的能力证据链 |
| Resources（资源中心） | 提供设备知识库、用户手册入口、交付标准和问题排障资料 | 个人资料检索与知识整理入口 |
| Business Tools（业务工具） | 提供工单、报价、询价和招投标等本地工作记录工具 | 浏览器本地保存的工作辅助记录 |

各模块不是彼此独立的展示页。例如，方案中的产品组合引用 Product Lab 的产品主数据，方案的验证范围引用 Labs 中已有的实验记录，Dashboard 再聚合这些真实存在的内容和状态。

## 能力形成链路

```text
技术学习
  ↓
产品理解
  ↓
实验验证
  ↓
厂商分析
  ↓
方案设计
  ↓
售前流程
  ↓
能力证据
```

1. **技术学习**：建立网络、安全、云计算和解决方案设计的基础知识结构。
2. **产品理解**：从客户为什么需要产品出发，理解产品定位、能力和使用边界。
3. **实验验证**：把技术判断转换为可复现的检查目标、方法和预期观察结果。
4. **厂商分析**：以客户场景和技术匹配为依据比较路线，不做厂商排名。
5. **方案设计**：将客户问题映射到技术能力、产品组合、架构关系和建设阶段。
6. **售前流程**：串联需求分析、技术选型、方案设计、验证准备与实施规划。
7. **能力证据**：通过已有产品模板、实验记录、方案模板和厂商分析说明当前积累，而不是使用无依据的评分。

## 关键工程实践

### 数据复用

站点以 `technologyDomains`、`productLabs`、`labExperiments`、`solutionCases` 和 `vendorComparisons` 作为主要内容来源。首页和 Dashboard 聚合这些已有数据，避免为展示页重复维护产品、实验、方案或厂商列表。

### 状态一致性

不同模块保留各自业务语义，例如实验可以处于“进行中”且已经建立模板，方案可以处于“正在完善”且已经建立架构。展示层使用明确的中文状态，不把模板建立或架构建立误写成项目完成。

### 模块引用关系

Solutions 只保存产品 ID 和方案中的使用原因，产品名称与定位从 Product Lab 数据读取；验证关联只引用 Labs 的实验 ID，不复制实验状态。这种引用方式减少重复内容，也让无效引用能够被单独容错处理。

### 真实性控制

内容将学习材料、个人验证记录和模板方案与真实客户交付明确区分：

- 实验验证项描述验证目标、检查方式和预期观察结果，不虚构实际测试结论。
- 方案使用“模板方案”“正在完善”等状态，不声明已上线、已交付或已验收。
- 厂商分析不包含排名、市场份额、价格优势或未经验证的性能结论。
- 产品内容不绑定未经核验的型号、版本、授权、命令和性能参数。

### 静态部署

项目使用 HTML、CSS 和原生 JavaScript 实现，通过 Node.js 构建脚本生成静态产物，并由 Vercel 从 GitHub `main` 分支自动构建部署。项目没有第三方前端运行时依赖，业务工具数据仅保存在当前浏览器的 Local Storage 中。

### 可访问性优化

站点在现有原生组件基础上补充了键盘焦点、筛选状态、结果播报、移动导航状态、弹窗可访问名称、表单标签和装饰元素隐藏等处理。Hash 路由支持直接访问以及浏览器前进、后退，同时保留响应式布局和移动端导航。

## 项目限制

- 当前只有 1 条实验记录，且验证证据仍需在个人可控环境中继续补充。
- 当前只有 1 个方案，属于正在完善的模板方案，不是已完成的真实客户案例。
- 项目没有真实客户名称、预算、上线日期、验收结果、项目收益或性能提升数据。
- 产品学习模板已经建立统一结构，但具体版本、授权、兼容性和配置细节仍需以厂商最新官方资料为准。
- 厂商对比用于练习场景化选型，当前覆盖范围和资料深度有限，不能替代正式项目中的产品测试与选型论证。
- 工单、报价、询价等业务工具只使用浏览器本地存储，不具备账号、云同步、权限控制和多人协作能力。
- 项目仍需持续增加真实的个人实验记录、验证证据和来源清晰的学习资料。

这些限制也是后续建设的边界：项目优先保证内容可以说明来源和当前状态，再逐步增加深度，不用虚构数量或成果填充页面。
